# 2026-07-01 — Review consolidation kickoff

## Summary

Opened the review-consolidation project the day after Warden landed, to make explicit that Warden is
the canonical automated reviewer and the other review tools complement it rather than compete.

## Work completed

- Created the project: `plan.md` (thesis, role map, five phases) and this progress note; indexed it
  under Infrastructure & hygiene in `docs/projects.md`.
- Documented the division of labor in `CONTRIBUTING.md` ("Code review" section): Warden owns the
  inline comments posted to a PR; the built-in `/code-review` and `/security-review` are a local
  inner-loop pass run _without_ `--comment`; `/review` (Standards + Spec) stays as the holistic
  on-demand review.

## Decisions

- **Warden is the path forward** — the single automated gate. Overlap is resolved in Warden's favor;
  complements keep only the ground Warden doesn't cover.
- **Nothing to delete.** `/code-review` and `/security-review` are built-in Claude Code commands,
  not repo files — this is a convention change, not file removal.
- The consolidation's real payoff is Phase 3: encoding the repo's own standards as custom Warden
  skills, so the gate enforces house rules (not just generic correctness) and can eventually subsume
  `/review`'s Standards axis.

## Next steps

- Land the Warden secrets ([#302](https://github.com/DavidJFelix/DavidJFelix/issues/302)) and verify
  the gate on a real PR (Phase 1).
- Draft the first repo-standard Warden skill (Phase 3) — likely the file-naming / kebab-case rule or
  the bun-not-bash scripting rule, since both are mechanical and high-signal.

## Open questions

- Always-on vs. gate behind a `Warden` label — depends on real per-PR cost and noise, unknown until
  the gate runs live.
- Which house rules are worth a dedicated Warden skill vs. left to Biome/Oxlint/actionlint (Warden
  should not re-check what a deterministic linter already gates).
