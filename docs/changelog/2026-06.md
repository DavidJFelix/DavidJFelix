### fix(f311x): restore production — free domains from a leftover dev worker; smoke-test the chat loop

f311x had been serving a non-hydrating dev-mode page since the 2026-06-06 deploy. #221
(merged 2026-06-11) fixed the ingress, deploy-toolchain, and artifact bugs, but every prod
deploy afterward still failed at custom-domain reconciliation: a leftover dev-stage worker
(`f311x-website-dev-davidfelix-*`) still held `f311x.com` / `www.f311x.com`, and Cloudflare
won't attach one hostname to two Workers. Detaching the two domains from that worker let the
prod deploy (CD run 27504856773) claim ingress and ship the correct production bundle —
f311x.com and www now hydrate and the chat echo streams end to end. The #220 token-scope
worry was a red herring: the deploy authenticated fine and needed no zone scope. Also
hardened `bin/smoke-test.ts` to POST the chat endpoint and require the AG-UI echo stream
back, so a hydrated shell with a dead backend now fails the CD gate too (it previously
checked only hydration).

### fix(repo): let self-hosted Renovate run `mise lock`

The Renovate dependency dashboard was warning "mise lock was requested to run, but
mise is not permitted in the allowedUnsafeExecutions". Renovate's mise manager wants
to refresh our `mise.lock` files (`.config/mise.lock`, `Advent-of-Code/2023/mise.lock`)
by running `mise lock`, which it classifies as an unsafe execution and blocks unless
explicitly allowlisted. `allowedUnsafeExecutions` is a self-hosted-only option, so it
can't go in `.github/renovate.json`; set `RENOVATE_ALLOWED_UNSAFE_EXECUTIONS='["mise"]'`
in the `cron-renovate` workflow instead.

### chore(cd): deploy workflows for the five scaffolded sites

The five scaffolded domain sites (onvibes.org, revision.city, startchi.com,
monicandavid.com, pkg.dog) had CI but no deploy job; each now has a `cd-deploy-*`

CD template — pinned actions, mise, frozen `pnpm install`, then `wrangler deploy` via
the shared `CLOUDFLARE_API_TOKEN` / `CLOUDFLARE_ACCOUNT_ID` secrets. Triggers are
path-filtered to each app dir plus the workflow's own file. Closes the deploy gap left
when the apps were scaffolded.

## 2026-06-13

### feat(pkg.dog): move from Vue SPA to Nuxt; settle type-checker policy

`apps/pkg.dog` now runs on Nuxt 4 (Nitro `cloudflare_module` preset) instead of a bare
Vue 3 Vite SPA, giving it a server story like the repo's other meta-frameworks. PandaCSS
is wired through Nuxt's `postcss` config (there is no `@pandacss/nuxt` package), Ark UI
Vue stays as the component library, and the landing page is a single `app/app.vue`.
`nuxt build` emits the worker to `.output/server/index.mjs` with assets in
`.output/public`, matched by `wrangler.toml`; `wrangler deploy --dry-run` validates the
bundle and `ASSETS` binding.

This also settles the repo's type-checker policy: each framework uses its own native
checker, and tsgo (`@typescript/native-preview`) is reserved for the case where the
alternative would be bare `tsc`. So Nuxt uses `nuxt typecheck` (vue-tsc), and
`apps/monicandavid.com` (SvelteKit) was switched back from tsgo to `svelte-check`. The
TanStack Start / React apps keep tsgo (no framework-specific checker), and Astro keeps
`astro check`. `vue-tsc` is pinned to 3.3.4 to stay clear of pnpm's 24h release-age
cooldown without adding a `minimumReleaseAgeExclude` block.

### feat(djf.io): Pagefind search with Cmd/Ctrl+K dialog

inline integration in `astro.config.mjs` runs Pagefind over the built output

### fix(djf.io): footer top border rendered near-white instead of zinc.800

The footer paired the `borderTop: '1px solid'` shorthand with the `borderColor`
shorthand. Panda emits each as its own atomic class, and `.bd-t_1px_solid` landed
after `.bd-c_zinc.800` in the stylesheet, so the shorthand reset `border-top-color`
back to `currentColor` -- the body's near-white `zinc.100` text color -- making the
bar above the copyright line glow bright against the dark theme. The header only
looked right because its `border-bottom` class happened to extract before the color
class. Switched both header and footer to the `borderBottomColor` / `borderTopColor`
longhands, which Panda deterministically sorts after shorthands, so the token color
always wins regardless of extraction order. Verified via computed styles in
Playwright: both borders now resolve to `rgb(39, 39, 42)`.

### fix(calendar-visualizer): correct holiday detection; first real tests

Writing the app's first unit tests surfaced two bugs in the calendar state
logic. (1) Holiday probing used local-midnight `Date` instants against
date-holidays ranges computed in US time, so on any machine at/east of UTC
(including CI runners) every holiday rendered one day late. (2)
`getHolidays(day)` returns the whole year's holidays, not that day's, so any
holiday-flagged day carried all ~11 holiday names. Fixed by fetching each
year's holidays once and matching by `YYYY-MM-DD` calendar date. The logic
moved from `calendar.tsx` into a new `calendar-state.ts` (no PandaCSS/React
imports) with 10 co-located vitest cases covering month slotting, spacer
padding, weekend/holiday tagging, and partial intervals — verified under UTC,
New York, Los Angeles, and Tokyo timezones. `passWithNoTests` removed now that
the suite exists.
