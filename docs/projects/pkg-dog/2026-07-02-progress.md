# pkg.dog — 2026-07-02 progress

## Session summary

Built the Phase 2 feasibility spike and got a yes for the happy path: `@std/collections@1.3.0`
decomposes into 50 independently installable parts that all verify against upstream, and a spike
registry serves them over the npm protocol -- a stock `npm install` resolves parts and their
cross-part dependencies through it. Also closed out Phase 1 by making the landing explain the
focusing lens concretely.

Mid-session course correction from David: pkg.dog is _its own package manager_ (like JSR), not a
publisher onto npm. Consumers will write `pkgdog:@scope/package` with potentially finer module
specification. The spike's npm-publish step was replaced with the registry server accordingly.

## Decisions

- **Granularity**: per-export parts with real dependencies, not undirected components. An export
  that imports another export depends on that part (exact version) with specifiers rewritten in the
  emitted files; internals reached by several parts become shared `__-internal-*` parts;
  mutually-cyclic entries merge into one part.
- **Barrels are skipped, not decomposed into parts**: a re-export-only entry (like root `mod.js`) is
  the thing being decomposed; consumers of a part import the part.
- **Distribution model (corrected in-session)**: pkg.dog is its own registry, the JSR playbook --
  not packages published onto npm. `pkgdog:@std/collections/chunk` maps reversibly to the npm-compat
  name `@pkgdog/std__collections__chunk` served from pkg.dog's endpoint, so stock npm/bun/pnpm
  clients work unchanged. The spike proves the serve half locally (`mise run verify-registry`).
- Parts reuse the upstream version verbatim for now; the real registry needs its own version epoch.

## Next steps

- Move the registry onto the pkg.dog Worker: same npm protocol, parts in R2/KV, decompose-on-demand
  or pre-warmed.
- Build the `pkgdog:` resolver CLI (like `jsr add`): turns `pkgdog:@scope/package/part` into the
  scope mapping + npm-compat dependency in the consumer's manifest.
- Harden the spike toward a design doc: a second, messier target package (es-toolkit or zod) to
  exercise shared internals at scale; `tsc`-level type resolution verification; AST-based specifier
  rewriting; version epoch scheme.

## Blockers / open questions

- Feasibility beyond stdlib-shaped packages (heavy shared internals, deep graphs, type-only
  entanglement) is the next thing the spike must answer -- the happy path is proven, the messy
  middle is not.
- The `__` mangling separator is ambiguous if an upstream name itself contains `__` (JSR accepts the
  same ambiguity); revisit before the registry goes public.
