## Part 3 — CD (alchemy deploy from Actions)

Wired `.github/workflows/cd_deploy_f311x.yml`: on push to main (path-filtered), `pnpm install` then
`alchemy deploy --stage prod --yes`, authenticating with the existing `CLOUDFLARE_API_TOKEN` /
`CLOUDFLARE_ACCOUNT_ID` repo secrets — the same ones wrangler uses for the other apps.

Confirmed from the Alchemy CLI + source before writing it:

- Auth is env-based (no `~/.alchemy` profile needed); no `ALCHEMY_PASSWORD` since the stack has no
  secrets.
- State lives in Cloudflare (`Cloudflare.state()`), so CI reconciles against it.
- `Cloudflare.Vite` builds the client + server bundle itself during deploy
  (`createBuilder().buildApp()`) — so the workflow needs no separate build step, and must not run a
  plain `vite build` (that skips Alchemy's Cloudflare plugin).
- `alchemy deploy` defaults the stage to `dev_${USER}` (→ `dev_runner` in CI), so the workflow pins
  `--stage prod` for a stable production deployment.

Not yet run for real: needs the workflow on `main` (or a dispatch) plus the live secrets. The first
run creates the prod-stage resources (R2 buckets, AI Gateway, the Worker) and assumes the token
carries R2 + AI Gateway + Workers permissions. The local manual deploy used the default
`dev_${USER}` stage, so the CI `prod` deployment is a separate, CI-owned stack.
