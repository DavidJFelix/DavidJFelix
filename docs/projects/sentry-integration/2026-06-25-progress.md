# 2026-06-25 ŌĆö Full-fleet rollout

Wired Sentry into the remaining ten apps on branch `claude/pensive-euler-w2a96w`, templatizing the
djf.io pilot. One commit per app; everything ships **dark** (no code runs until a DSN is provided).
Supersedes the earlier, simpler attempt on `claude/fervent-fermi-tdsnzu` (direct-to-vendor SDKs) ŌĆö
this rollout uses the refined same-origin **tunnel** (`/bugs`) so ad/tracker blockers can't drop
real-user errors.

## Decisions

- **Refined relay everywhere** (per the user): every app gets the `/bugs` tunnel, not direct
  `*.ingest.sentry.io` POSTs. The static Astro sites and the ravrun SPA gained a server (Cloudflare
  adapter / a small worker) to host it.
- **Client SDK per framework**: `@sentry/astro` (3 Astro sites), `@sentry/react` (4 TanStack Start +
  ravrun), `@sentry/browser` (SvelteKit, Nuxt).
- **Server-side capture, tiered**: the four TanStack Start apps and ravrun wrap their fetch handler
  with `@sentry/cloudflare`'s `withSentry` (workerd-native, no `@sentry/node`) reading a runtime
  `SENTRY_DSN` worker var. The 3 Astro sites stay client-only (prerendered; djf.io precedent).
  **Deferred** on SvelteKit (monicandavid) and Nuxt (pkg.dog): their server SDKs are
  `@sentry/node`-based and won't run on workerd, and both are low-traffic placeholders ŌĆö the tunnel
  already captures client errors. Revisit with `@sentry/cloudflare` if SSR visibility is needed.
- **Gating invariant**: the browser SDK is gated on the inlined build-time `*_SENTRY_DSN` literal
  (Astro `PUBLIC_`, Vite `VITE_PUBLIC_`) so Vite/Astro dead-code-eliminates it in keyless builds ŌĆö
  verified each keyless client bundle ships **zero** Sentry code. pkg.dog is the exception: Nuxt
  `runtimeConfig` (`NUXT_PUBLIC_*`) is a runtime gate, so its SDK is a lazy chunk loaded only when
  the var is set.

## Per-stack work

- **f311x** (TanStack/Alchemy): `src/server.ts` wraps `@tanstack/react-start/server-entry`;
  `wrangler.toml main` points Alchemy's `Cloudflare.Vite` at it. `vite build`/smoke use the
  unwrapped default bundle; prod (alchemy) uses the wrapper.
- **forzamonica / revision.city / startchi** (TanStack/@cloudflare-vite-plugin): same
  `src/server.ts` wrap via `main`.
- **ravrun** (SPA): new `src/worker.ts` (relay + `ASSETS` fallback) wrapped with `withSentry`.
- **3 Astro sites**: `sentry.client.config.ts` + the integration, `enabled.server: false`; the
  tunnel rides the adapter worker.
- **monicandavid / pkg.dog**: client `@sentry/browser` + the tunnel; server deferred.

Per app: typecheck, lint, format, unit (relays + config 100%-covered), build, and a `wrangler dev` /

org/project.

## Worker vars are set by CD, not by hand

The runtime `SENTRY_DSN` worker var is now bound by the deploy workflow from the **same**
`SENTRY_DSN_<APP>` repo variable that feeds the client build ŌĆö no separate Cloudflare dashboard step,
and no drift (a `wrangler deploy` overwrites a Worker's plaintext vars each run, so a hand-set var
would be wiped on the next deploy anyway):

- **forzamonica / revision.city / startchi / ravrun**: `wrangler deploy --var SENTRY_DSN:${{ vars.SENTRY_DSN_<APP> }}`.
- **f311x**: declared in `alchemy.run.ts` (`env: {SENTRY_DSN: process.env.SENTRY_DSN ?? ''}`), fed from
  the `alchemy deploy` step's env. Resolves the old open question about whether a manually-set var
  survives `alchemy deploy`.
- **pkg.dog**: `wrangler deploy --var NUXT_PUBLIC_SENTRY_DSN:ŌĆ” --var NUXT_PUBLIC_POSTHOG_KEY:ŌĆ”` from
  two new repo variables (`SENTRY_DSN_PKG_DOG`, `POSTHOG_KEY_PKG_DOG`), since Nuxt reads these from
  `runtimeConfig` at runtime rather than inlining them at build.

Empty until activation, so this stays dark (`withSentry` no-ops on an empty DSN).
