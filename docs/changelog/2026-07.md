### feat(forzamonica.com): grow the storefront site map and shared layout

The storefront's map was four routes (home hero, `/products`, `/products/$handle`, `/cart`) with a
two-link header and no footer. This pass fills out the pages a shop needs before launch and the
chrome that binds them: an `/about` placeholder (brand direction is still open, so the copy says
so), three policy stubs (`/policies/shipping`, `/policies/returns`, `/policies/privacy`) sharing a
`PolicyArticle` shell, and a styled 404 via the root route's `notFoundComponent` -- SSR'd with a
real 404 status, verified by e2e. Catalog stays deliberately flat: collections routes were
considered and deferred until the product line exists.

Layout-wise the root route gained a proper `component` (header / flex-grow main / footer inside the
existing shell), a new three-column `SiteFooter`, an About link in the header, and a cart-count
badge fed by a root-route loader calling a new badge-sized `fetchCartQuantity` server function -- no
cookie means no Shopify round trip, and every cart mutation already ends in `router.invalidate()`,
which re-runs the loader and keeps the badge honest. Every page now sets its own title (the PDP from
loader data).

The e2e suite grew from 2 to 6 tests (footer sitemap, about, policies, 404 status + chrome) with
header/footer-scoped selectors now that link names repeat, and the home visual baseline was
re-authored for the new chrome. `playwright.config.ts` gained a `PLAYWRIGHT_CHROMIUM_EXECUTABLE`
escape hatch for sandboxes that ship a system Chromium instead of allowing the pinned download; CI
is unaffected.

## 2026-07-01

### fix(ci): stop preview workflows duplicating their sticky PR comment

The preview-comment upsert only read the first page of a PR's comments
(`GET /issues/{n}/comments?per_page=100`). That endpoint returns comments oldest-first and has no
sort option, so once a PR crossed ~100 comments the sticky comment fell off page 1, the marker was
never found, and every rerun posted a fresh duplicate. `bin/comment-preview.ts` now walks every
page, collecting all marker-bearing comments; it edits the oldest in place and deletes the rest,
which both fixes forward and cleans up duplicates earlier runs already left on open PRs. Duplicate
DELETEs treat a 404 as a no-op so concurrent runs racing on the same stale comment don't abort the
cleanup.

Consolidated the two copies while here: `apps/f311x/bin/comment-preview.ts` was a near-identical
fork, so `cd-preview-f311x.yml` now calls the shared root script (via `$GITHUB_WORKSPACE`) with
`PREVIEW_MARKER` / `PREVIEW_TITLE` / `PREVIEW_FOOTER`, matching how the `preview-wrangler` composite
already drives it. The new `PREVIEW_FOOTER` input preserves f311x's Alchemy-stage teardown wording;
the wrangler apps keep the default version phrasing. One script, one place to fix, all 11 preview
workflows covered.

The vendored mattpocock skills already followed the pattern -- solid copies in `.agents/skills/`,
tracked symlinks in `.claude/skills/` -- but the repo-authored surfaces did not. Moved
`dependency-freshness`, `panel-review`, and the six review personas into `.agents/` and left

known upstream regressions and are avoided. No configuration alternative exists -- Claude Code has
no setting to relocate skill/agent discovery, and the plugin route would impose `plugin:name`
namespacing. Conventions recorded in `AGENTS.md`.

### fix(f311x): repair the wip github.ts CI-credentials stack

The wip Alchemy stack pinned Stack's generics as `<void, ConfigError>`, but Stack's second type
parameter is the provider requirements, not the error channel (ConfigError is already hardwired in
