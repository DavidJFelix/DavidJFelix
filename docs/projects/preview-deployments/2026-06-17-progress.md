# 2026-06-17 — Phase 2: generalize previews to the wrangler apps

## Summary

Rolled the per-PR preview pipeline from f311x (Phase 1, Alchemy) to all nine wrangler-deployed
apps. Each PR now uploads a non-active preview version, smokes the preview URL, screenshot-diffs it
against a committed baseline, and posts a sticky comment. Built on a shared composite action +
repo-root `bin/` scripts so the nine apps share one implementation.

## Work completed

- **Shared infra** (`feat(preview): shared wrangler preview infra + calendar-visualizer`):
  - `bin/upload-preview.ts` — `wrangler versions upload --preview-alias pr-<N>`; resolves the
    deterministic `pr-<N>-<worker>.<subdomain>.workers.dev` URL by reading the account subdomain out
    of wrangler's output. Pure `parsePreviewUrl` is unit-tested (`bin/upload-preview.test.ts`, run by
    `mise run test:bin`, gated in `ci-repo.yml`).
  - `bin/smoke-url.ts` — generic URL smoke (200 + complete HTML doc + first hashed asset serves),
    the remote sibling of each app's `bin/smoke-local.ts`.
  - `bin/comment-preview.ts` — sticky preview comment, generalized from f311x (per-app marker/title).
  - `.github/actions/preview-wrangler/action.yml` — build → upload → smoke → Playwright → comment.
  - Thin per-app `cd-preview-<app>.yml` callers (one per app; no teardown job).
- **Apps wired** (one commit per group): calendar-visualizer, djf.io, ravrun, davidjfelix.com,
  onvibes.org, revision.city, startchi.com, monicandavid.com, pkg.dog. Each got a `playwright.config.ts`
  with the `PREVIEW_URL` seam, a committed Linux/chromium `toHaveScreenshot` baseline, a `test:e2e`
  mise task, and (custom-domain apps) `preview_urls = true`.
- **Local verification** for every app: `mise run build`, generate the baseline, a clean `test:e2e`
  to confirm stability, `bin/smoke-url.ts` against a real local boot, and the app's full check.

## Decisions

- **Mechanism: `wrangler versions upload --preview-alias`, not a per-PR named worker.** Upload never
  applies triggers, so it can't hijack the custom domains and needs no teardown — strictly simpler
  and safer than the named-worker alternative. (Validated against Cloudflare docs during planning.)
- **Composite action over a reusable `workflow_call` workflow** — matches the existing
  `setup-mise` precedent and runs in the app directory cleanly.
- **Screenshots on every app** (David's call, refined from "where Playwright exists"). Near-static
  single-page apps get one full-page baseline; djf.io reuses its existing behavioral suite.
- **SSR boots**: TanStack Start → `vite preview` (workerd via `@cloudflare/vite-plugin`);
  SvelteKit/Nuxt → `wrangler dev` on the built worker. Each Playwright config mirrors the app's
  `bin/smoke-local.ts`. SvelteKit/Nuxt e2e specs live in a top-level `e2e/` dir (outside the
  framework source roots and the vitest include).
- **forzamonica.com deferred** to the forzamonica-shop project — its storefront pages depend on
  external mock.shop data (unstable baseline) and that project is mid-flight.
