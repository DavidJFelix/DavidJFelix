# 2026-06-21 — client-side Sentry wired fleet-wide

Took the rollout from kickoff to "wired, pending activation." Client-side Sentry error monitoring is
now in all eleven apps, no-op until a DSN is set.

## Work completed

- Each app gained an `observability/` module: a pure, unit-tested `config.ts` (`resolveSentry` — off
  unless a DSN is present) and a `client.ts` that **dynamically imports** the SDK only when enabled,
  so it never enters any SSR/worker bundle and a credential-less build never fetches it.
- SDK by archetype: `@sentry/react` for the React / TanStack Start apps (f311x, forzamonica.com,
  revision.city, startchi.com) + the ravrun SPA; `@sentry/browser` for the static Astro sites
  (djf.io, calendar-visualizer, davidjfelix.com, onvibes.org), SvelteKit (monicandavid.com), and
  Nuxt (pkg.dog).
- Init is client-only per framework: `getRouter()` behind `!import.meta.env.SSR` (TanStack), the SPA
  entry (ravrun), a hoisted `<script>` (Astro), `onMount` (SvelteKit), a `.client.ts` plugin (Nuxt).
- DSN var by framework convention: `VITE_PUBLIC_SENTRY_DSN` (Vite/TanStack/SvelteKit),
  `PUBLIC_SENTRY_DSN` (Astro), `NUXT_PUBLIC_SENTRY_DSN` (Nuxt).
- Verified per app: typecheck, lint, unit, build, and smoke where present; confirmed no `@sentry/*`
  code in the server/worker bundles.

## Decisions

- **Conservative SDK choice.** Used `@sentry/react` / `@sentry/browser` (client) rather than the
  framework meta-SDKs. The official guides pull in server pieces that break Cloudflare's workerd
  runtime — `@sentry/tanstackstart-react`'s worker export resolves to `@sentry/node`
  (getsentry/sentry-javascript#20038), which crashes the Worker with no useful diagnostic.
- **Dynamic-import + no-op-without-DSN.** Lets the instrumentation land and ship now, with David
  flipping it on later by setting a DSN — the code is inert until then.
