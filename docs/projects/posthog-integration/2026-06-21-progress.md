# 2026-06-21 — client-side PostHog wired fleet-wide

Took the rollout from kickoff to "wired, pending activation." Client-side PostHog product analytics
is now in all eleven apps, no-op until a project key is set. Done back-to-back with the Sentry
rollout, sharing the same `observability/` module and env wiring.

## Work completed

- Each app gained an `observability/` module: a pure, unit-tested `config.ts` (`resolvePostHog` —
  off unless a key is present; defaults the host to `https://us.i.posthog.com`) and a `client.ts`
  that **dynamically imports** `posthog-js` only when enabled, so it never enters any SSR/worker
  bundle and a credential-less build never fetches it.
- `posthog-js` is uniform across every app (no framework-specific PostHog SDK needed). Init is
  client-only via the same per-framework entry points as Sentry (TanStack `getRouter` SSR guard, SPA
  entry, Astro hoisted `<script>`, SvelteKit `onMount`, Nuxt `.client.ts` plugin).
- Key/host vars by framework convention: `VITE_PUBLIC_POSTHOG_KEY` / `…_HOST` (Vite/TanStack/
  SvelteKit), `PUBLIC_POSTHOG_*` (Astro), `NUXT_PUBLIC_POSTHOG_*` (Nuxt). Init uses
  `defaults: '2026-01-30'`, PostHog's forward-compatible config knob (autocapture + SPA pageviews).
- Verified per app: typecheck, lint, unit, build, and smoke where present; confirmed no `posthog`
  SDK code in the server/worker bundles.

## Decisions

- **Global init, not the React provider (yet).** Autocapture + pageviews come from `posthog.init`
  alone; the `@posthog/react` `usePostHog` provider was skipped to keep the SDK out of the SSR/worker
  bundle and the wiring uniform across five frameworks. Explicit `posthog.capture` calls can be added
  per app later from client-only code.
- **`core-js` build-script decision.** `posthog-js` pulls `core-js`; each app's
  `pnpm-workspace.yaml` now records `allowBuilds: core-js: false` (its postinstall is a funding
  notice; the JS ships prebuilt), so installs don't abort under pnpm 11.
