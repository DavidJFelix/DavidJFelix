## 2026-06-18

### fix(djf.io): standard.site document.site must be the publication AT-URI

A document record's `site` field must be the publication's AT-URI
(`at://{did}/site.standard.publication/{rkey}`); a bare url like `https://djf.io` is rejected by
the validator as malformed. The prior cut wrote `site` as the publication url to key ownership off
the url, but that produced invalid records. The sync now writes `site: publicationUri()` (the
AT-URI), and the build resolver scopes documents to djf.io by first finding the publication whose
`url` is `https://djf.io` (so ownership is still anchored on the url, not the record key) and then
matching documents whose `site` equals that publication's AT-URI. Other publications sharing the
repo stay excluded.

### fix(djf.io): standard.site records must use TID record keys

The `site.standard.publication` and `site.standard.document` lexicons require TID record keys
(13-char base32-sortable; validity is purely syntactic), so the first cut's human-readable keys —
`self` for the publication and the post slug for documents — were rejected by the standard.site
validator. Publication keys are now a committed server-assigned TID constant (`PUBLICATION_RKEY`),
so `publicationUri()` stays synchronous and offline. Document keys are server-assigned: the sync
script `createRecord`s new documents (the PDS assigns the TID) and matches existing ones by their
`path` to update in place, and the site resolves each post's `at://` URI from the PDS at build time
(`src/lib/standard-site-resolve.ts`, one memoized fetch). That resolution is gated behind a CD-only
`STANDARD_SITE_RESOLVE` flag and degrades to warn-and-omit, so local/CI/e2e/preview builds stay
offline and deterministic (the document `<link>` is omitted; the publication link is always present
via the constant). The CD deploy runs the sync before `astro build` so records exist when the build
resolves them. The sync is non-destructive (create/update only); migrating or pruning legacy
records is a manual step. The reconcile decision is a pure, unit-tested planner; the networked
resolver is excluded from lib coverage. Documents are scoped to djf.io by their `site` field (the
one publication whose url is `https://djf.io`), so another publication sharing the repo is never
matched or resolved. Re-validate at site-validator.fly.dev.

### feat(djf.io): standard.site / AT Protocol compatibility; close standard-site-compat

Made djf.io a first-class long-form publisher on the AT Protocol (the network behind Bluesky)
and closed `docs/projects/standard-site-compat/`. A shared, framework-agnostic
`src/lib/standard-site.ts` holds the public DID (`did:plc:nlbldots3jn3lk6mzca4rqzm`, verified
`alsoKnownAs: at://djf.io`), publication metadata (reusing the RSS strings), and the rkey/URI
helpers. Record keys are deterministic — publication rkey `self`, document rkey = the post
slug — so every `at://` URI is computable from that one DID; the static build emits the same
URIs the sync writes, with nothing persisted back to the repo.

Website artifacts: two `.well-known` endpoints — `site.standard.publication` (the publication
AT-URI) and `atproto-did` (the bare DID, an HTTP handle-verification path for `@djf.io` that
complements the DNS `_atproto` record) — plus `<link rel="site.standard.*">` tags on the base
layout and every post, covered by unit (100% lib coverage) and Playwright e2e tests.

Go-live: `bin/sync-standard-site.ts` (bun, `mise run sync-standard-site`) logs in to bsky.social
with an app password, asserts the session DID before any write, and idempotently upserts one
publication plus one document per post via `putRecord`. The CD workflow gained a guarded sync
step between `astro build` and `wrangler deploy`, gated on `ATPROTO_APP_PASSWORD` (an encrypted
Actions secret, #249) so it skips when unset and otherwise blocks the deploy on sync success —
the site never ships referencing records that don't exist. Running `mise run` in that job also
required `MISE_TRUSTED_CONFIG_PATHS`, matching `ci-djf-io.yml`. The app password grants full
write to the repo; it is secret-stored, rotatable, and fenced by the session-DID guard.
Post-deploy confirmation is the standard.site validator at site-validator.fly.dev.

### feat(preview): per-PR preview deployments for every wrangler app; close preview-deployments

Generalized the per-PR preview pipeline from f311x (its Alchemy-based Phase 1 proving ground —
preview deploy + smoke + screenshot on every PR) to the nine wrangler-deployed apps:
calendar-visualizer, djf.io, ravrun, davidjfelix.com, onvibes.org, revision.city, startchi.com,
monicandavid.com, and pkg.dog. Each PR touching an app now uploads a non-active preview version,
smokes the preview URL, screenshot-diffs it against a committed baseline, and upserts a sticky
comment with the URL.

The wrangler shape is simpler than f311x's Alchemy stage. `wrangler versions upload --preview-alias
pr-<N>` uploads a non-active version with a deterministic
`pr-<N>-<worker>.<subdomain>.workers.dev` URL; because it never applies triggers, production routes
and custom domains stay on the active version — no route-hijack risk and nothing to tear down
(Cloudflare prunes old aliases), so there is no `closed`/destroy job unlike f311x. Custom-domain
apps set `preview_urls = true` in their `wrangler.toml` so the alias URL resolves.

The nine apps share one implementation: a composite action `.github/actions/preview-wrangler/`
(build → upload → URL smoke → Playwright screenshots → sticky comment) invoked by thin per-app
`cd-preview-<app>.yml` callers, plus three repo-root bin scripts — `bin/upload-preview.ts` (the
versions-upload + deterministic-URL resolver, with a unit-tested parser gated by a new
`mise run test:bin` job in `ci-repo.yml`), `bin/smoke-url.ts` (the remote sibling of each app's
`bin/smoke-local.ts`: 200 + complete HTML document + first hashed asset serves), and
`bin/comment-preview.ts` (generalized from f311x). A composite action was chosen over a reusable
`workflow_call` workflow to match the existing `setup-mise` precedent and run in the app directory.

Screenshots run on every app, not just where Playwright already existed: each gets a committed
Linux/chromium `toHaveScreenshot` baseline and a `PREVIEW_URL` seam that points the suite at the
deployed preview in CI and at a local production boot when authoring baselines. The local boot
mirrors each app's smoke — `astro`/`vite preview` for the static and TanStack Start apps (the latter
serving the SSR worker in workerd via `@cloudflare/vite-plugin`), and config-driven `wrangler dev`
for the SvelteKit and Nuxt apps; djf.io reuses its existing behavioral Playwright suite via the same
seam.

Live-deploy verification is pending on the first real PR (the upload path needs CI secrets), and the
preview/smoke/screenshot results feeding the dependency-freshness auto-merge bar are tracked in that
project. With the infrastructure complete, this closes out `docs/projects/preview-deployments/`.
forzamonica.com is deferred to the forzamonica-shop project (its storefront pages depend on external
mock.shop data, which complicates a stable baseline).

### fix(repo): scope the mise CI cache key to the lockfile via a shared `setup-mise` action

`jdx/mise-action`'s default cache key hashes *all* mise config files repo-wide — its
`{{file_hash}}` globs `**/mise.toml` — so editing any app's `mise.toml` (even just adding
a task) invalidates the one shared key for every workflow. The #247 smoke-task commit did
exactly that, so every CI job started on a cold cache; dozens of parallel jobs then
stampeded the same key, and two unlucky ones (monicandavid `svelte-check`, pkg.dog
`format check`) stalled on the cold Node-tarball download until their 10-minute timeout
cancelled them. Factored the mise setup into a local composite action
`.github/actions/setup-mise` that pins the cache key to the real tool inputs —
`mise-v1-{{platform}}-${{ hashFiles('.config/mise.toml', '.config/mise.lock') }}` — and
converted all 79 `jdx/mise-action` usages across 28 workflows to it. App `mise.toml` edits
no longer bust the cache; only genuine tool-version bumps (which change `.config/mise.lock`)
do. Also added `.github/actions/**` to `ci-actions-lint.yml`'s triggers so the shared
action is linted when it changes. The 11 `cd-deploy` jobs (previously `cache: false`, a
template default with no recorded rationale) now use the shared cache too — safe because
the key is lockfile-scoped (identical pinned tool binaries) and deploys run only on
`push: main`, a trusted context whose cache scope no untrusted PR can write to.
