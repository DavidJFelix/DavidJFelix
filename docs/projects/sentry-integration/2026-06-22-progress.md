# 2026-06-22 — djf.io client-side Sentry

## Summary

Started the Sentry rollout on djf.io. Established that djf.io is a **fully static site** and
therefore gets **client-only** instrumentation (errors + performance tracing); the backend/Worker
half is deferred to apps that actually have a runtime (f311x leads, per the plan).

## Decisions

- **djf.io is client-only.** Every route is build-time: the blog and tag pages use `getStaticPaths`,
  and `og/[...slug].png`, `rss.xml`, and the two `.well-known` endpoints are all deterministic from
  build-time content with no per-request input. Nothing needs a runtime, so there is no backend to
  instrument. Standing up the Cloudflare adapter just to host a server-side Sentry would manufacture
  a backend the site doesn't need.
  - Aside: the OG route couldn't run on `workerd` as written anyway — it uses `node:fs` (font
    loading) and `sharp` (a native addon; native `.node` modules never run on Workers). Prerendering
    is strictly better here: built once in Node where `sharp` works, served as cached static images.
- **Feature set: errors + performance tracing.** No Session Replay — heaviest bundle, little to
  replay on a static blog.
- **Backend Sentry goes elsewhere.** f311x (real chat Worker) is the Worker proving ground; the
  server pattern (`@sentry/cloudflare` `withSentry`) lands there, not here.

## Work completed

- Added `@sentry/astro` (v10, Astro-6 compatible). Approved `@sentry/cli`'s build in
  `pnpm-workspace.yaml` (`allowBuilds`) so deploy-time source-map upload works.
- `sentry.client.config.ts`: `Sentry.init` with `browserTracingIntegration`, full trace sampling,
  disabled unless a DSN is present.
- `astro.config.mjs`: `sentry()` integration. Server SDK never injected (`enabled.server: false`);
  browser SDK bundled only when `PUBLIC_SENTRY_DSN` is set; source maps uploaded only when
  `SENTRY_AUTH_TOKEN` + `SENTRY_ORG` + `SENTRY_PROJECT` are all present; build telemetry off.
- `cd-deploy-djf-io.yml`: the production `astro build` step references the GitHub-managed Sentry
  config (`PUBLIC_SENTRY_DSN` from a repo variable, the auth token from a secret, org/project from
  variables) in a step-scoped env. Chosen over a committed literal to match how the repo already
  manages deploy config (`CLOUDFLARE_API_TOKEN`, `ATPROTO_APP_PASSWORD`).
- Net effect: local, CI, preview, and e2e builds ship **zero** Sentry code and stay deterministic;
  production (DSN set) gets errors + tracing.
- Verified: build (no-DSN and DSN paths), Biome format, lint, typecheck, unit (100%), and Playwright
  e2e (25 passed) all green.
