# onvibes.org — 2026-07-02 progress

## Session summary

Built the first slice of the builder toolchain: a `/chat` page (React island) backed by a Flue agent
running in the deployed Cloudflare Worker. Architecture: **Flue owns the Worker and hosts Astro
inside it** — `src/app.ts` (Hono) mounts Flue's agent API at `/api` and forwards everything else to
the prebuilt Astro worker (`dist/server/entry.mjs`). PR #303.

## Work completed

- Added `@astrojs/react` + a `/chat` page mounting `src/components/Chat.tsx` (`@flue/react` +
  `@flue/sdk` against `/api`). Extracted `src/layouts/BaseLayout.astro` so head boilerplate, PostHog
  analytics, and the Panda base stylesheet are shared by every page.
- Added `src/agents/assistant.ts`: a keyless faux echo agent. Two registrations are required —
  `registerFauxProvider` (pi-ai transport) **and** Flue's `registerProvider('onvibes', ...)`
  (model-specifier resolution); the faux responder re-queues itself because the provider consumes
  one queued response per model call.
- Two-phase build: `build:astro` (panda + astro) then `build:worker` (`flue build` +
  `bin/patch-flue-wrangler.ts`, which re-injects the `[assets]` binding Flue's build drops). Astro
  reads a DO-free `wrangler-astro.jsonc` (adapter `configPath`); `wrangler.toml` carries the Durable
  Object migrations for Flue.
- Pointed CD (deploy + preview) at `dist-flue/onvibes_org/wrangler.json` — previously both deployed
  the Astro-only worker, so `/api` 404'd on the live preview. Smoke now boots the Flue worker,
  checks `/` and `/chat`, and POSTs the agent endpoint requiring the echo (verified green locally);
  local e2e boots the same config.

## Decisions

- **Flue-hosts-Astro** (not Astro-hosts-Flue): Flue's Cloudflare build generates the Durable
  Objects; `@astrojs/cloudflare/hono`'s `cf()` requires Astro's build to own the worker, so instead
  the self-contained Astro entry is imported and forwarded to.
- **Faux echo stays** until a real model is wired; swap point is documented in `assistant.ts`.
- Adding an agent under `src/agents/` requires a matching `new_sqlite_classes` migration entry in
  `wrangler.toml` (Flue generates `Flue<Name>Agent`).
- **Isolated per-PR preview workers** (f311x's stage pattern, generalized):
  `wrangler versions upload` cannot carry DO migrations (error 10211) and would share production's
  DO storage, so previews now `wrangler deploy` a separate `onvibes-org-pr-<N>` Worker (routes
  stripped, workers.dev URL) and delete it on PR close. Reusable pieces for future DO apps:
  `.depot/actions/preview-worker` + `preview-worker-teardown`, backed by
  `bin/deploy-preview-worker.ts` / `bin/teardown-preview-worker.ts`.
